| [startup-latency.md](plans/startup-latency.md) | two fixes shipped 2026-07-22; one lead open | **Why the glasses start rendering seconds after the phone screen.** Measured against the reference Unity app (+2.76 s to `GfxThreadStart` vs our +5.39 s). Fixed the bootstrap retry backoff (a flat 60-frame wait for an Activity that arrived in 145 ms) → 4.24 s. Records that **an autoload cannot fix the rest** — autoload `_ready()` beats the main scene but autoload `_process()` does not, so the first bootstrap attempt stays pinned to scene-load completion. Then found the SDK's 1.39 s input start was **our `input_source=3`** making it call `SetHandTrackingEnabled` synchronously (878 ms) for hand tracking the One Pro does not have -> now `xreal/input_source`, default Controller. 5.39 s -> ~3.9 s. Open: the ~1.45 s pin to main-scene load. |

Also: how to get pixels on the CPU for OpenCV, and why the Cb/Cr split was measured and rejected. |

| [fpv-streaming-plan.md](plans/fpv-streaming-plan.md) | device-verified (video + mic audio); **app audio blocked engine-side** | First-person-view H.264 streaming via the SDK's `libmedia_codec` HW encoder (`HWEncoder*` flat exports). `XrealSystem.stream_start/stream_push_frame/stream_stop`; `demo/stream_manager.gd` renders a head view into a SubViewport and feeds its GL texture to the encoder (local mp4, or live RTP). | [camera-zero-copy-investigation.md](archive/camera-zero-copy-investigation.md) | 2026-07-21: camera copy-path load PoCs, measured then reverted (code lives at `c5b9a67`). R8 `AHardwareBuffer` rejected by this gralloc (`isSupported R8=0, RGBA8=1, YCbCr420=1`). **Its headline conclusion — "the floor is the SDK acquire" — was disproved the same day** by disassembly and measurement (the two getters cost 4 µs and 0 µs); the section is struck through in place so the reasoning is not repeated. Still reliable: the AHardwareBuffer/gralloc survey and the misc findings. |
| [codex-headlock-analysis.md](archive/codex-headlock-analysis.md) | Led to the head-lock fix: `UpdateDeviceState(updateType=1)` refreshes the compositor render pose. Solved 2026-07-13. |
| [roll-tracking-investigation.md](archive/roll-tracking-investigation.md) | Roll was missing from the session-manager `NrPose`; solved by switching the eye cameras to the display pose (`GetHeadPoseAtTime` 4×4). |
| [multiview-investigation.md](archive/multiview-investigation.md) | Handoff written while head-lock was unsolved (it is now). Multiview was later fixed (2026-07-17): both eyes render; stays opt-in (no perf gain over Multipass). |
| [codex-input-start-analysis.md](archive/codex-input-start-analysis.md) | 2026-07-22: **why XR input start took 1.39 s for us and 0.20 s for Unity.** 878 ms of it is one synchronous `NativePerception::SetHandTrackingEnabled(true)` that `InputManager::InputStart @ 0x78794` makes only because we passed `input_source=3`. Rules out 6DoF SLAM (Unity uses the same tracking type), our hand-built `IUnityXRInput` table (input start only calls `SetDeviceConnected`, ignoring the return), and any sleep/timeout constant in the wrapper. |
| [codex-camera-acquire-analysis.md](archive/codex-camera-acquire-analysis.md) | 2026-07-21: **RE of the RGB-camera acquire path** (`libXREALXRPlugin.so` + `libnr_api.so` + `libnr_rgb_camera.so`). No zero-copy exists through the public C ABI — the plane pointer is a wrapper-owned pooled `std::vector`, the backend is a plain V4L2 MMAP ring (no `VIDIOC_EXPBUF`), and the one MediaCodec decoder is byte-buffer output, not a surface. Refutes camera-zero-copy-investigation.md's "floor is the SDK acquire": both getters are hash lookups and pointer arithmetic (measured 4 µs and 0 µs). Also IDs `TryGetRGBCameraFrame` as a destructive new-frame flag, and XREAL's own `nreal-ai/uvc_android` as the public reference for the capture layer. |
| [codex-multiview-analysis.md](archive/codex-multiview-analysis.md) | Multiview swapchain registration analysis. Multiview now works (opt-in); Multipass stays the default. |
| [codex-righteye-analysis.md](archive/codex-righteye-analysis.md) | Early root-cause hypothesis for the Multiview black right eye (`libnr_api` can't import a client `GL_TEXTURE_2D_ARRAY`) — disproved 2026-07-17: the real causes were two Adreno GLES layer-copy quirks, fixed in `src/gl.rs` (`0fe8f8e`). |
| [codex-stub-callbacks-analysis.md](archive/codex-stub-callbacks-analysis.md) | 2026-07-16: ruled out the stubbed Unity callbacks (`RegisterTextureProvider` / `GetPlatformData` / `RegisterDisplayProvider` / gfx device-event) as the Multiview layer-1 blocker (still valid). Its `libnr_api`-import verdict was disproved 2026-07-17 — Adreno layer-copy quirks were the real cause. |
| [codex-floatingmanager-analysis.md](archive/codex-floatingmanager-analysis.md) | 2026-07-16: refuted "floating return button not feasible", implemented + device-verified it, then **removed it in favour of Picture-in-Picture** (the button returned to the app but didn't keep the glasses rendering). RE record retained. |
| [codex-background-render-analysis.md](archive/codex-background-render-analysis.md) | 2026-07-16: **why the glasses freeze when our app is backgrounded.** Root cause: our NR submit is gated by Godot's `process()` on the display-0 window, whose SurfaceView is destroyed on `onPause` (`EGL_BAD_SURFACE`). The reference Unity app keeps rendering via **SurfaceView reparent onto the resumed glasses-display Activity** + pause-suppression (not a Service — that hypothesis is disproven). Fix = port that (Godot Android-template patch); substantial — ultimately shipped via auto-enter PiP instead (see background-render-plan.md). Corrects the old "multiResume keeps rendering after Home" over-claim. |
